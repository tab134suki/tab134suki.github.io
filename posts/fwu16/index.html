二维码离线传输2.0

jiyu1342025-09-202025-09-20

前言

在今年2月份的时候，我有尝试使用二维码来离线传输图片，虽然这是一个不错的想法，但是在当时并没有取得成功（属于是一个差一点就成功）。
不过好在我又进行了一次尝试，我选择了另一个道路来完成这个项目。
并且，这个2.0版本得到了升级，现在，它不仅可以适用于图片，更覆盖所有文件。
出于篇幅问题，有些地方我就不放源代码了，如果想看我与AI的对话，可以看末尾的链接。

想法？

我本来计划沿用之前的python来完成，因为这种方法相较于C语言较简单，但是…在我换了电脑后，我不知道在旧电脑都放了什么库进去，而且原本的文件中有一些错误，包括像二维码导出问题，识别问题。
我对此头皮发麻，这些问题比较难以处理…
所以我在放弃的前一秒选择了另一个赛道——HTML
其实在使用它的时候我又在想，html可不可以实现我的想法？我起初有些害怕。但在我用AI生成的时候，居然成功了！
我很惊讶，赶忙对这个项目进行了跟进处理。

详细信息

  1. 核心技术：

    • HTML/CSS/JavaScript: 核心前端技术。
    • Web APIs:
      • FileReader API: 读取用户上传的文件。
      • Web Crypto API: 实现加密解密（我们将使用对称加密 AES-GCM 替代 ECC，因为 Web Crypto 对 ECC 的支持更偏向于密钥协商 ECDH）。
      • getUserMedia API: 调用摄像头。
      • Canvas API: 用于二维码生成和扫描。
      • jsQR Library: 在浏览器中扫描二维码。
      • qrious or kjua Library: 在浏览器中生成二维码。
      • Cache API 或 IndexedDB: 存储生成的二维码图片和接收的文件（更推荐 IndexedDB，因为它更强大且持久）。
      • Blob URL: 为存储在内存或 IndexedDB 中的文件创建临时下载链接。
    • Web Workers (可选): 对于大文件的加密/解密或二维码生成，可以放到 Web Worker 中执行，避免阻塞主线程。
    • 模块化 (ES6 Modules): 组织代码，使其更清晰。

  2. 发送端 (Sender) 流程:

    • 用户选择文件（多个）。
    • 前端 JavaScript 读取文件，计算 SHA-256。
    • 将文件内容转换为 Base64。
    • 按用户指定的块大小分割 Base64 字符串。
    • 为本次传输生成一个随机的 AES-GCM 密钥。
    • 使用 AES-GCM 密钥加密每个数据块。
    • 按照文档格式拼接每个块的数据字符串（包含文件名、大小、类型、SHA256、索引、总数）。
    • 将密钥（用一个固定的“主密钥”加密，或者直接放在第一个二维码里）和第一个加密块的数据字符串组合，作为第一个二维码的内容。后续二维码只包含加密块数据字符串。
    • 使用 JavaScript 库（如 qrious）将每个数据字符串生成二维码图片（<canvas> 或 data:image URL）。
    • 将生成的二维码图片（或其数据）和原始文件信息存储到 IndexedDB。
    • 在网页上以幻灯片形式展示二维码。

  3. 接收端 (Receiver) 流程:

    • 用户点击“开始扫描”按钮，调用摄像头。
    • 使用 jsQR 持续扫描视频流中的二维码。
    • 解析扫描到的二维码数据。
    • 如果是第一个二维码，从中提取 AES-GCM 密钥（并解密，如果之前被加密）。
    • 使用 AES-GCM 密钥解密每个扫描到的块。
    • 解析解密后的数据，提取文件名、内容、索引等信息。
    • 将解密后的块按索引存储在内存或 IndexedDB 中。
    • 在 UI 上显示扫描进度（已扫描/总数）。
    • 计算并显示扫描速度。
    • 当所有块都接收完毕后：
      • 按索引顺序拼接所有块的 Base64 内容。
      • 对拼接后的 Base64 进行解码，得到原始文件的 ArrayBuffer 或 Blob。
      • 计算重组后文件的 SHA-256 并与元数据中的进行比对（可选，增加可靠性）。
      • 将重组后的文件 Blob 存储到 IndexedDB。
      • 在页面上提供一个下载链接（使用 URL.createObjectURL(blob)）。

  4. 文件存储与下载:

    • IndexedDB: 使用浏览器的 IndexedDB API 来持久化存储：
      • 生成的二维码图片数据 (Base64 或 Blob)。
      • 原始文件的元数据和 Base64 内容（发送端，用于重新生成）。
      • 接收端解密并重组后的文件 Blob。
    • 下载: 当用户需要下载文件时，从 IndexedDB 读取文件 Blob，然后使用 URL.createObjectURL(blob) 创建一个临时的下载链接。
